# Backend API

YAPAT exposes a RESTful API built on **FastAPI**. All endpoints communicate using JSON, and every authenticated request requires a valid bearer token.
Identifiers such as `{id}` are opaque strings generated by the backend.

This page summarizes the main backend resources and their relationships:

* **Users** authenticate and belong to one or more **Teams**.
* **Teams** own **Datasets**.
* **Datasets** contain **Recordings**, which are segmented into **Snippets**.
* **Snippets** receive **Annotations**, supporting multi-label entries.
* **Training Examples** and **Classifiers** enable similarity search and weakly supervised learning.
* A unified **Feed** endpoint provides sampling-based snippet retrieval.
* **Taxonomy** endpoints connect species and environmental terms to authoritative sources such as GBIF and ENVO.

---

## Authentication and Users

YAPAT implements **bearer-token authentication** (JWT) and optional **invite codes** for team membership.
Authentication uses standard REST conventions.

```text
Authorization: Bearer <token>
```

### User model

```jsonc
User {
  "id": "user-123",
  "email": "user@example.org",
  "name": "Optional Name",
  "created_at": "2025-11-10T21:00:00Z"
}
```

### Endpoints

#### `POST /auth/register`

Register a new user.

#### `POST /auth/login`

Authenticate and receive an access token.

#### `GET /me`

Return the authenticated user and their team memberships.

## Teams and Memberships

Teams are the basic collaboration unit.
Each user can belong to multiple teams, and each team can manage multiple datasets.

### Team model

```jsonc
Team {
  "id": "team-1",
  "name": "My Research Group",
  "created_at": "2025-11-10T21:00:00Z",
  "created_by": "user-123"
}
```

### Membership model

```jsonc
TeamMember {
  "user_id": "user-123",
  "team_id": "team-1",
  "role": "owner"   // or "member"
}
```

### Endpoints

* `POST /teams` — create a new team.
* `GET /teams` — list teams the user belongs to.
* `GET /teams/{team_id}` — retrieve a specific team.
* `POST /teams/{team_id}/join` — join a team via invite code.

## Datasets

Datasets represent collections of recordings grouped under a team.
YAPAT indexes existing audio directories rather than storing raw audio itself.

### Dataset model

```jsonc
Dataset {
  "id": "ds-1",
  "team_id": "team-1",
  "name": "BR-Mamiraua-2025",
  "description": "Mamirauá pilot deployment",
  "source_uri": "/path/to/mamiraua2025/",
  "created_at": "2025-11-10T21:05:00Z"
}
```

### Endpoints

* `GET /teams/{team_id}/datasets`
* `POST /teams/{team_id}/datasets` — registers the dataset, scans recordings, and generates snippets.
* `GET /datasets/{dataset_id}` — retrieve dataset metadata.

## Snippets

Snippets are short segments (typically 3 s) cut from recordings according to a **SnippetConfig**.

### SnippetConfig model

```jsonc

  "hop_duration_sec": 1.5,
```

## Embeddings and Embedding Models

Embeddings are fixed-length numeric vectors derived from snippets by a trained model.

### Principles

* Embedding vectors are **never returned** via REST.
* Operations like `similarity`, `initialize_model`, and `refine` rely on stored vectors.
* Snippets must match the model’s expected duration and sample rate.

### EmbeddingModel

```jsonc
EmbeddingModel {
  "id": "birdnet_v2",
  "name": "BirdNET v2",
  "description": "3-second windows at 48 kHz, 1024-dim embedding",
  "input_duration_sec": 3.0,
  "sample_rate_hz": 48000,
  "embedding_dim": 1024,
  "created_at": "2025-11-10T21:30:00Z"
}
```

### SnippetEmbedding

```jsonc
SnippetEmbedding {
  "snippet_id": "snip-123",
  "embedding_model_id": "birdnet_v2",
  "version": 1,
  "created_at": "2025-11-10T21:35:00Z"
}
```

### Endpoints

* `GET /embedding_models`
* `GET /snippets/{snippet_id}/embeddings`

## Annotations

Annotations assign taxa to snippets.
Taxa are referenced by **namespaced identifiers** (e.g., `gbif:2420576`).

### Annotation model

```jsonc
Annotation {
  "id": "ann-101",
  "dataset_id": "ds-1",
  "snippet_id": "snip-123",
  "taxon_id": "gbif:2420576",
  "resolved_name_snapshot": "Hyla versicolor",
  "confidence": 0.92,
  "created_at": "2025-11-10T21:10:00Z",
  "created_by": "user-42"
}
```

### Endpoints

* `GET /snippets/{snippet_id}/annotations`
* `GET /datasets/{dataset_id}/annotations` — supports filters: `taxa`, `annotators`, `from`, `to`, `limit`.
* `POST /snippets/{snippet_id}/annotations` — batch create.
* `DELETE /annotations/{annotation_id}`

**Notes**

* `taxon_id` must be resolvable via `/taxonomy/search`.
* The resolved name is snapshotted for reproducibility and not updated automatically.
* Legacy `gbif_taxon_key` and `scientific_name` fields are deprecated.

## Feed Generation

The **Feed API** retrieves snippets according to a sampling strategy.

```text
GET /datasets/{dataset_id}/feed
```

### Common parameters

| Parameter         | Type   | Description                                                             |
| ----------------- | ------ | ----------------------------------------------------------------------- |
| `sampling_method` | string | e.g., `random`, `similarity`, `initialize_model`, `validate`, `refine`. |
| `limit`           | int    | Max number of snippets (default 50).                                    |
| `cursor`          | string | Pagination token.                                                       |
| `status`          | string | Optional filter by snippet status.                                      |

Feed items include `snippet`, `recording`, `annotation_summary`, and method-specific `scores`.
See **[Feed Methods](feed-methods.md)** for detailed method behavior.

## Classifiers

Classifiers are **lightweight manifests** bundling independent, per-taxon linear heads trained on snippet embeddings.
Each classifier belongs to a **dataset** and one **EmbeddingModel**.
Predictions are multilabel, one-vs-rest.

### Classifier model

```jsonc
Classifier {
  "id": "clf-2025-01",
  "dataset_id": "ds-1",
  "name": "Frogs v1",
  "embedding_model_id": "birdnet_v2",
  "target_taxa": ["gbif:2420576", "gbif:2498293"],
  "status": "ready"   // "training" | "ready" | "failed"
}
```

**Notes**

* `target_taxa` uses namespaced IDs (`gbif:*`, future `envo:*`).
* Classifiers are **snapshots** that reuse existing heads and train missing ones asynchronously.
* Access control derives from the dataset.
* Provenance and metrics live at the internal head level.

### Endpoints

#### `POST /classifiers`

Create a classifier manifest.

```json
{
  "dataset_id": "ds-1",
  "name": "Frogs v1",
  "embedding_model_id": "birdnet_v2",
  "target_taxa": ["gbif:2420576", "gbif:2498293"],
  "annotation_filters": {
    "min_confidence": 0.8,
    "annotators": ["user-42"],
    "snippet_status": "labeled"
  },
  "extra_positive_training_example_ids": ["te-1", "te-2"]
}
```

**Response**

```jsonc
{
  "id": "clf-2025-01",
  "dataset_id": "ds-1",
  "name": "Frogs v1",
  "embedding_model_id": "birdnet_v2",
  "target_taxa": ["gbif:2420576", "gbif:2498293"],
  "status": "training"
}
```

* Returns immediately; training proceeds in background.
* Once complete, `status` = `"ready"`.

#### `GET /classifiers`

List classifiers accessible to the user.

#### `GET /classifiers/{classifier_id}`

Retrieve classifier metadata.

#### `DELETE /classifiers/{classifier_id}`

Delete or deactivate a classifier manifest.

```jsonc
  "dataset_id": "ds-1",
  "embedding_model_id": "birdnet_v2",
  "snippet_config_id": "cfg-default-3s",
  "taxon_id": "gbif:2420576",
  "version": 1,
  "weights_uri": "modelstore://.../w_b.bin",
  "artifact_hash": "sha256:…",
  "trained_at": "2025-11-11T10:05:00Z",
  "trained_by": "user-123",
  "status": "ready",
  "metrics": {
    "train_precision": 0.91,
    "train_recall": 0.84,
    "train_f1": 0.87
  },
  "training_summary": {
    "annotation_filters": { "min_confidence": 0.8, "annotators": ["user-42"] },
```

## Taxonomy Validation and Consistency

YAPAT uses **namespaced taxon identifiers** (e.g., `gbif:2420576`) as authoritative keys.
Names are resolved automatically and snapshotted for reproducibility.

### Write-time rules

* `taxon_id` must resolve via `/taxonomy/search`.
* `scientific_name` from clients is ignored for authority.
* On success, the backend stores `resolved_name_snapshot`.
* Invalid IDs yield `400 Bad Request ("invalid_taxon_id")`.

### Snapshot policy

* Snapshots (`resolved_name_snapshot`) are **not updated** when taxonomies change.
* Clients may refresh via `/taxonomy/resolve`.

### Endpoints

* `GET /taxonomy/search?q={query}` — text search.
* `GET /taxonomy/resolve?id={taxon_id}` — resolve one ID.
* `POST /taxonomy/resolve` — batch resolve multiple IDs.

### Consistency guarantees

* Every stored annotation or training example has a **resolvable** `taxon_id`.
* Snapshots guarantee reproducibility.
* Future extensions will support multiple namespaces (GBIF, ENVO, etc.).
